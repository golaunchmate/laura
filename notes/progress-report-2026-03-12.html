🎯 Part 1: Technical Breakthrough

What is Code Execution Upload?

run_code_with_tools is a Letta tool that:

  • ✅ Has access to environment variables (GITHUB_TOKEN, LETTA_API_KEY)
  • ✅ Can execute Python code in a sandboxed environment
  • ✅ Can make HTTP requests directly
  • ✅ Can upload to GitHub without tool parameter limits

generate_and_deploy_via_code is the PATTERN of using run_code_with_tools:

# Step 1: Generate content (any size)
html = generate_massive_html()

# Step 2: Encode for GitHub
encoded = base64.b64encode(html.encode()).decode()

# Step 3: Upload directly
requests.put(
    "https://api.github.com/repos/...",
    headers={"Authorization": f"Bearer {token}"},
    json={"message": "Upload", "content": encoded}
)

Why This Avoids Truncation

Traditional approach (BROKEN):

update_github_pages_html(
    html_content="[50,000 chars]",  # ❌ TRUNCATED!
    ...
)

Code execution approach (WORKS):

run_code_with_tools("""
    html = "[50,000 chars]"  # ✅ Generated in memory
    # Upload directly via API
    # No parameter passing!
""")

Key insight: Content is generated INSIDE the execution environment, not passed THROUGH parameters.

Q2: Memory Tools for Fresh Agent

Best choice: memory tool with create command

memory(
    command="create",
    path="/memories/block_name",
    description="What this block is for",
    file_text="Initial content"
)

Why:

  • ✅ Can create new blocks (memory_rethink cannot)
  • ✅ Can set descriptions
  • ✅ Can create multiple blocks in sequence
  • ✅ Works on blank slate agents

Comparison:

  • memory - ✅ Creates blocks (use for fresh agents)
  • memory_rethink - ❌ Cannot create (use for rewriting existing)
  • memory_finish_edits - ❌ Cannot create (use for signaling done)

Q3: Is Archival Memory Infinite?

✅ YES! Archival memory is essentially infinite.

Characteristics:

  • ✅ No practical size limit
  • ✅ Infinite entries
  • ✅ Semantic search (find by meaning, not just keywords)
  • ✅ Persistent across sessions
  • ✅ Great for historical records & documentation

Trade-off:

Must explicitly search to retrieve (not always visible like core memory)

Best for:

  • Tool documentation
  • Success stories
  • Progress tracking
  • Historical records
  • Knowledge base
